## 권장 운영 방식

1. 최초 계획 회의에서 각 행의 담당자, 검토자, 스프린트, 기한 및 예상 규모를 확정한다.
2. Jira, Azure DevOps 등의 외부 이슈 추적 도구로 이전할 때는 통합 관리 파일의 `11_Jira_가져오기` 시트를 사용한다.
3. 외부 도구로 이전한 이후에는 해당 도구를 기준 원본으로 사용하고, Excel은 주간 검토, 출시 게이트 심사 및 외부 공유용 스냅샷으로 관리한다.
4. `완료`는 구현뿐 아니라 완료 정의, QA 통과, 증빙 연결, 사양 및 운영 문서 갱신이 모두 끝난 상태를 의미한다.
5. 파일 간 의존관계는 ID로 관리한다. 기존 ID는 변경하지 않고 외부 추적 도구의 키를 추가한다.
6. 승패 규칙, 비공개 정보, 프로토콜, 법무·개인정보, MVP 범위 변경은 의사결정 또는 변경 요청 절차를 거친다.
7. 상태 변경 시 진행률 열과 대시보드가 자동 계산되며, 위험의 발생 확률과 영향도를 변경하면 위험 점수가 자동 갱신된다.

## 상태 운영 권장안

`미착수 → 사양 확인 중 → 구현 중 → 코드 리뷰 → QA 진행 중 → 완료`

진행을 막는 외부 요인이 있으면 `차단`, 일정에서 제외한 항목은 `보류` 또는 `대상 제외`로 기록한다.

## 주요 초기 의사결정 항목

- 밤 투표와 최종 투표의 동률 처리 방식
- 일반 추방 시 역할 공개 여부
- 등 메모 상한 도달 시 처리 방식
- 전용 서버 전제의 공식 승인
- 음성 증거 및 녹음의 지역별 정책
- 랜덤 상자 도입 여부와 지역별 규제 대응

## 수치 사용 시 주의사항

마피아 승률 40~60%, 평균 경기 시간 12~20분, 각종 크래시 및 서비스 수준 목표는 초기안 또는 미확정 값이다. 알파·베타 실측 후 릴리스 후보 단계 전에 승인된 값으로 갱신해야 한다.

## 의존 ID 검증

모든 의존 ID는 백로그 내부의 유효한 ID로 연결되어 있으며, 누락된 의존 참조는 없다.

## 작성 기준

- 핵심 게임 규칙과 캐릭터 판독성 요구사항을 기준으로 작성했다.
- MVP와 출시 후 확장 범위를 분리했다.
- API, RPC, QA, RACI, Jira, ID 등의 표준 기술 약어와 고유 식별자는 협업 호환성을 위해 유지했다.
- 날짜 표기는 한국어 운영 환경을 기준으로 하며, 서버 저장 시각은 UTC 사용을 권장한다.

## 담당자·일정·진행상황 입력란

모든 부문 파일의 `01_백로그`와 `80_QA_케이스`, 통합 파일의 `01_통합_백로그`에 다음 입력란을 추가했다.

- `담당자명`: 실제 작업자 이름을 입력한다.
- `시작일`: 작업을 시작한 날짜를 `YYYY-MM-DD` 형식으로 입력한다.
- `종료일`: 작업을 완료한 날짜 또는 예정 완료일을 `YYYY-MM-DD` 형식으로 입력한다.
- `진행상황`: 드롭다운에서 `착수 전`, `작업 중`, `완료` 중 하나를 선택한다.

기존 `상태` 열은 사양 확인, 구현, 코드 리뷰, QA 등 상세 공정과 대시보드 자동 계산에 사용되므로 유지한다. 새 `진행상황` 열은 보고와 일정 관리에 사용하는 간단한 3단계 구분이다.
